tools定位：
一个word文档的格式调整工具，输入一个格式比较混乱的原始word文件  输出一个符合格式要求的调整后word文档

#设计架构
分三步骤：
step1:解析输入文档
-解析规则：

【全局约束（适用前置规则和所有模块）】
1. 表格和图片内的所有内容：保留原始格式，不做任何处理，不进入前置规则和四模块判断
2. 括号兼容全角半角：( = ( 或 （，) = ) 或 ）
3. 小数点/句点兼容全角半角：. = . 或 ．
4. 编号内外的空白容错：所有模块正则在前导、编号内部及编号与正文之间均允许任意长度的空白（含全角/半角空格）
5. 加粗判定粒度：段落内任意 run 加粗即视为加粗段落；各模块内部的"加粗"条件指匹配模式（编号/括号/数字开头）所在的 run 本身加粗即命中，无需整段加粗
6. 加粗前缀提取容错：段首若存在空文本或纯空白的 run（常因 Word 编辑残留出现且 bold 属性可能为 None），跳过这些 run 继续向后寻找首个加粗 run，并保留其原始空白文本，避免误判为"非加粗"

【前置规则（优先于四模块）】
段落内无任何 run 加粗 → 直接套用模板 Body 全量格式，不进入四模块判断。

【模块执行顺序（含任意加粗 run 的段落进入）】
模块四 → 模块一 → 模块二 → 模块三，匹配到即停止。模块三为兜底。

【章节边界定义】
以 H1 为章节分隔点；当文档没有 H1 时，首个出现的 H2-H5 标题同样开启章节上下文，使依赖"当前章节内"的模块二在无 H1 文档中仍可工作。

解析模块一：编号模式匹配（主流程）
编号模式所在 run 加粗即命中。

编号模式                              | 判定层级
第N章 / 第一章（阿拉伯或汉字数字）    | H1
中文序号 + 、                         | H2
（一）类汉字序号（括号全角半角）      | H3（无需检查前文）
单级数字 + .（全角半角）              | 在上一个标题中向上回溯、跳过同型的"单级数字"标题，找到首个非同型标题；若该标题为（一）类 → H4；否则 → H3
两级数字（1.1 / 2.3等）              | H4
三级数字（1.1.1等）                  | H5

注：当识别为 Hj（j = n+1, n+2, ...）但预设只到 Hn 时：保留原始文本格式，仅将字体改为 Hn 的字体。

解析模块二：括号类编号（上下文依赖）
括号编号所在 run 加粗即命中。
“上一个标题”指当前章节内最近一个标题（含模块四拆分产生的标题）。无章节上下文时直接保留原始文本。
下移一层后若超出预设最大层级 Hn：保留原始文本格式，仅将字体改为 Hn 的字体。

case1：（数字/字母/罗马数字/希腊字母等，排除中文数字）（括号兼容全角半角）
即 (1) (2) (a) (b) (i) (α) 等系列
- 在当前章节内向上回溯，跳过同样匹配 case1 的同型标题，找到首个非同型标题作为"上一层标题"，在其层级基础上下移一层
- 当前章节内无可用上层标题 → 默认 H4
- 默认属于同一层级；当同章节内多种类型同时出现时，按 (1) > (a) > (i) > 其他 确定相对嵌套关系

case2：数字/字母/罗马数字/希腊字母等）（右括号兼容全角半角，无左括号）
即 1) 2) a) b) i) 等系列
- 在当前章节内向上回溯，跳过同样匹配 case2 的同型标题，找到首个非同型标题作为"上一层标题"
  - 上一层标题是 case1 → 在该 case1 层级基础上下移一层
  - 上一层标题不是 case1 → 查看其属于模块一中的哪一种，在其基础上下移一层
- 当前章节内无可用上层标题 → 默认 H5
- 默认属于同一层级；当同章节内多种类型同时出现时，按 1) > a) > i) > 其他 确定相对嵌套关系

解析模块三：兜底段落（加粗）
前提：不满足模块四、模块一、模块二任何匹配条件的加粗段落（含数字开头的漏网情况）。
统一处理：保留原始文本格式，仅将字体改为当前格式模板中 Body 的字体（从加载的 .yaml 模板 body.font_name 字段读取）。

解析模块四：数字开头 + 冒号 + 加粗混合段落
“数字开头”含阿拉伯数字和中文数字；数字开头所在 run 或冒号所在 run 加粗即触发。
- 内部复用模块一的正则匹配逻辑（独立执行）确定编号对应层级
- 若冒号前内容不符合模块一任何行（正则全部失配）：保留原始文本格式，仅修改字体
- 多个冒号时以第一个冒号为拆分点
- 统计第一个冒号之后的全部字符数：
  case1：> 34 字 → 冒号前为标题（层级由模块一正则定），冒号后为正文 Body，拆分为两段，不再重新走四模块；拆出标题以判定层级计入当前章节标题上下文
  case2：≤ 34 字 或冒号后直接换行/为空 → 全部加粗内容作为标题，层级由模块一正则定；同样计入当前章节标题上下文

step2: 前端人工审核
- Tab1：段落切片表
         原始文本 | 层级下拉（预填自动判定结果）| 置信度 | 命中规则
         低置信度行高亮，引导人工核验（目标 ≤5% 工作量）
- Tab2：原文预览（原始格式只读）
- Tab3：格式化预览
         已套用 YAML 模板的真实排版效果（字体/字号/缩进/行距全部呈现）
         Tab1 下拉变更后实时更新对应段落的排版效果
- 用户点击"确认输出" → 直接进入 step4

step3: 应用格式模板（内嵌于 step2 的实时渲染中）
- 格式模板从 .yaml 文件加载（支持多模板，路径可配置，不可修改内容）
- 格式规格参见根目录 report_cn.yaml（含 H1-H5/Body 的字体/字号/加粗/对齐/缩进/行距/页边距）
- Tab3 每次渲染 = step3 的实时执行
- 用户确认时的 Tab3 状态即为最终输出状态，无需重复计算

step4: 输出格式化文档
- 以 step2 确认时的层级 + YAML 格式为输入，直接写入文档
- .docx 输入 → 直接输出 .docx
- .doc 输入 → 解析前用 win32com.client 调用 Word COM 转为 .docx，处理完再转回 .doc 输出
- 表格/图片保留原始内容和格式不做处理

技术栈：
- 后端：Python + python-docx
- doc 转换：win32com.client（调用本机 Word COM，Windows 环境）
- API：FastAPI
- 前端：纯 HTML + JS
- Tab3 实时预览渲染：后端将段落按 YAML 格式转为带内联样式的 HTML 片段返回前端
