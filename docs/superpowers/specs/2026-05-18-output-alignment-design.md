# 输出文件格式与预览统一 + 图片表格强制居中

日期：2026-05-18
状态：设计已确认（含 code review 修订），待写实施计划

## 背景

当前项目（D:\word_format_2）输出 docx 时存在两类问题：

1. 文字段落的样式由 `formatter/applier.py` 写入，预览（Tab3）的样式由 `formatter/preview.py` 写入；两端各自从 template 字典取字段、各自有默认值，存在**潜在的字段口径漂移**：例如 `indent_first_line` 预览端默认 24、导出端默认 0；`space_before/after` 预览端根本未渲染。当前 `report_cn.yaml` 已为每级显式填齐字段，故差异未触发，但属随时可能爆发的隐患。
2. 图片所在段落被当成普通段落处理，会被强行写入正文的对齐与首行缩进，破坏原有居中；表格则原样保留，不会自动居中。前端 Tab3 已用红字提示「输出文件中的所有表格和图片格式都采用原文格式」，但当前导出实际未做到对图片段落「保留原貌」。

参考项目（D:\word_format）采取「补丁回写」策略：复制原文 → 仅对已分配段落写样式 → 保存。本设计沿用同一思路，并补齐图片/表格居中。

## 目标

1. **文字部分预览所见即输出所得**：预览端与导出端从同一份样式解析逻辑取值，杜绝字段口径漂移。所有 template 字段（含 `space_before` / `space_after` / `indent_first_line` 默认值）两端一致。涵盖普通分支（h1–h4 / body）与 Hj fallback 分支（仅改字体）。
2. **图片段落**：保留原段落一切属性，仅强制居中。判定与处理细节见下方「设计决策」。
3. **表格**：保留原文表格内部一切内容与格式，仅将顶层表格本身的水平对齐强制设为居中。无条件执行，不判定原状态。不递归处理嵌套表、不处理单元格内段落对齐、不处理单元格内的图片。

## 非目标

- 不引入参考项目的标题降级保护（H1→H3 自动降为 H2）。
- 不引入 TOC 提取。
- 不修改 doc↔docx 转换、页边距、前端、拆分段落逻辑、Hj 回退逻辑（仅统一其字段读取路径）。
- 不让预览 Tab3 渲染图片或表格（保持当前红字提示语义）。
- 不处理表格单元格内的图片（即使 cell 内含图，也不对其做居中）。`doc.paragraphs` 天然不枚举单元格内段落，因此无需写跳过逻辑。

## 设计

### 数据流

```
report_cn.yaml ──► load_template ──► 样式解析（单一真相源）
                                            │
                          ┌─────────────────┴─────────────────┐
                          ▼                                   ▼
                   预览端 (preview.py)                  导出端 (applier.py)
                          │                                   │
                          ▼                                   ▼
                      Tab3 HTML                          文字段落写样式
                                                              │
                                                              ├─ 含图片段落：整段保留原貌，仅强制居中
                                                              └─ 全部顶层表格：仅设水平居中
```

### 设计决策（含 code review 修订）

1. **单一真相源**：抽出独立的样式解析模块 `formatter/style_resolver.py`，输出包含所有字段（含默认值）的标准样式对象。预览端把它转成 CSS、导出端把它写到 docx 段落属性。两端使用完全相同的默认值集合。Hj fallback 分支同样走解析模块（返回「仅 font_name」标记），保证两端 fallback 行为口径一致。

2. **图片段落识别**：识别判据为段落 XML 中是否存在 `.//w:drawing` **或** `.//w:pict`（VML，覆盖 .doc 转 .docx 后老格式图片的常见情形）。

3. **图文混排段落处理策略**（采用方案 a）：只要段落含图片元素（drawing 或 pict），无视前端 detected_level、无视段内是否同时有文字 run，**整段一律保留原貌**——不改字体、不改字号、不改加粗、不改首行缩进、不改行距、不改段前段后。仅对该段落强制设为居中。
   理由：贴合 CLAUDE.md「表格和图片内容：保留原始格式，不做任何处理」的强约束；实现最简单；避免对 run 级别做判定带来的复杂度。

4. **居中的写入方式**：直接写段落 `w:pPr/w:jc = center`（OOXML 元素层），不走 `para.alignment` 高层 API。原因是若原段落对齐继承自 style 链，仅设 `para.alignment` 可能不覆盖继承值。表格居中同理，写在 `w:tblPr/w:jc`。

5. **表格居中范围**：仅遍历 `doc.tables` 顶层，**不递归进入单元格内嵌套表**，**不修改任何单元格内段落对齐**，**不处理单元格内的图片**。

6. **范围最小**：所有改动遵循「补丁回写」原则——只改需要改的属性，不重建结构。

## 文件改动范围

- 新增 `formatter/style_resolver.py`（约 40 行）
- 修改 `formatter/applier.py`：字段读取改走解析模块；新增图片段落识别函数（识别 `w:drawing` + `w:pict`）；新增段落级强制居中函数（直接写 `pPr/jc`）；新增表格居中函数（直接写 `tblPr/jc`，仅顶层）
- 修改 `formatter/preview.py`：字段读取改走解析模块（仅重构，无视觉变化）
- 修改 `main.py` 的 `/export`：循环中加图片段落分支（识别到图片即跳过样式应用，仅写居中，忽略前端 detected_level）；循环后调用表格居中
- 不动：前端、转换器、模板加载、解析器、规则、拆分逻辑、Hj 回退逻辑（仅其字段读取路径走 resolver）、页边距

## 验收标准

1. 同一份输入文档，预览 Tab3 的每个文字段落与导出 docx 中对应段落的字体名、字号、加粗、对齐、首行缩进、行距、段前、段后**完全一致**。该一致性涵盖 Hj fallback 分支（H5 等深层标题段，两端均仅套用最深层级字体）。
2. 输入文档中所有图片在输出 docx 中均居中显示，且其所在段落的字体、加粗、缩进、行距、段前段后**与原文完全一致**（不被预设样式覆写）。即使该段落同时包含文字 run，文字 run 也保留原貌。即使前端为该段落选择了某个 H 级别，导出仍走图片分支。
3. 输入文档中所有顶层表格在输出 docx 中均水平居中。表格内部内容、单元格对齐、嵌套表、单元格内图片均与原文一致，不做任何修改。
4. 拆分段落、Hj 回退（仅字体）、doc↔docx 转换、页边距等既有行为无回归。
