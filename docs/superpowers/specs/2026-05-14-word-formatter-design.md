# Word 文档格式化工具 — 设计文档

**日期：** 2026-05-14
**状态：** 待审阅

---

## 一、项目定位

输入格式混乱的 Word 文档（.doc / .docx），输出符合预设格式模板的整洁文档。核心价值是将人工格式整理工作量压缩到 5% 以下。

---

## 二、整体流程

```
上传文档
  ↓
step1: 解析 → 生成段落切片列表（层级 + 置信度 + 命中规则）
  ↓
step2: 前端审核（Tab1/Tab2/Tab3）→ 人工少量修正
  ↓                              ↑ Tab3 实时渲染 = step3
step3: 应用格式模板（内嵌于 step2 实时渲染）
  ↓ 用户点击"确认输出"
step4: 输出格式化文档
```

---

## 三、Step1：解析输入文档

### 3.1 全局约束

1. 表格和图片内容：保留原始格式，不进入任何判定逻辑
2. 括号兼容全角半角：`(` = `(` 或 `（`，`)` = `)` 或 `）`
3. 小数点/句点兼容全角半角：`.` = `.` 或 `．`
4. 加粗判定粒度：段落内任意 run 加粗即视为加粗段落；模块内"加粗"条件指匹配模式所在 run 加粗即命中

### 3.2 前置规则

段落内无任何 run 加粗 → 直接套用模板 Body 全量格式，不进入四模块。

### 3.3 模块执行顺序

**模块四 → 模块一 → 模块二 → 模块三**，匹配到即停止，模块三为兜底。

### 3.4 章节边界定义

以 H1 为章节分隔点。文档尚未出现 H1 时，所有依赖"当前章节内"的查找直接回退为保留原始文本。

### 3.5 模块一：编号模式匹配（主流程）

编号模式所在 run 加粗即命中。

| 编号模式 | 判定层级 |
|---|---|
| `第N章` / `第一章`（阿拉伯或汉字数字）| H1 |
| 中文序号 + `、` | H2 |
| `（一）` 类汉字序号（括号全角半角）| H3（无需检查前文） |
| 单级数字 + `.`（全角半角）| 当前章节内上一个标题为`（一）`类 → H4；否则 → H3 |
| 两级数字（`1.1` / `2.3`）| H4 |
| 三级数字（`1.1.1`）| H5 |

> 当识别为 Hj（j = n+1, n+2, ...）但预设只到 Hn 时：保留原始文本格式，仅将字体改为 Hn 的字体。

### 3.6 模块二：括号类编号（上下文依赖）

括号编号所在 run 加粗即命中。"上一个标题"指当前章节内最近一个标题（含模块四拆分产生的标题）。下移一层超出 Hn 时：保留原始文本格式，仅将字体改为 Hn 的字体。无章节上下文时直接保留原始文本。

**case1：`(1)` `(a)` `(i)` `(α)` 等（括号含内容，排除中文数字）**
- 查看当前章节内上一个标题属于模块一哪一种，下移一层
- 无上层标题 → 默认 H4
- 同章节多种类型共存时：`(1)` > `(a)` > `(i)` > 其他

**case2：`1)` `a)` `i)` 等（仅右括号，无左括号）**
- 上一个标题是 case1 → 在 case1 层级基础上下移一层
- 上一个标题不是 case1 → 查模块一类型，下移一层
- 无上层标题 → 默认 H5
- 同章节多种类型共存时：`1)` > `a)` > `i)` > 其他

### 3.7 模块三：兜底段落

前提：不满足模块四、模块一、模块二任何条件的加粗段落（含数字开头漏网情况）。

处理：保留原始文本格式，仅将字体改为模板 `body.font_name`。

### 3.8 模块四：数字开头 + 冒号 + 加粗（优先执行）

"数字开头"含阿拉伯数字和中文数字；数字开头所在 run 或冒号所在 run 加粗即触发。

- 内部独立复用模块一正则确定层级
- 冒号前内容正则全部失配 → 保留原始文本格式，仅修改字体
- 多个冒号以第一个为拆分点
- 统计第一个冒号之后的全部字符数：

| 字数 | 处理 |
|---|---|
| **> 34 字** | 冒号前 → 标题，冒号后 → Body，拆分为两段，不再走四模块；标题以判定层级计入章节上下文 |
| **≤ 34 字** 或冒号后为空/换行 | 全部加粗内容作为标题；计入章节上下文 |

### 3.9 输出数据结构

每个段落切片包含：

```json
{
  "index": 0,
  "original_text": "一、总体情况",
  "detected_level": "H2",
  "confidence": "high",
  "matched_rule": "模块一·中文序号+、",
  "is_split": false
}
```

置信度规则（high / medium / low）：
- **high**：仅模块一命中且无前文上下文依赖（不含单级数字 + `.` 行）；或模块四成功命中且内部正则命中的是无上下文依赖的模块一行
- **medium**：模块二命中；或模块一单级数字 + `.` 行；或模块四成功命中但内部正则命中的是上下文依赖行（如单级数字 + `.`）
- **low**：模块三兜底、模块四正则失配、无章节上下文回退

## 四、Step2：前端人工审核

### 4.1 Tab1 — 段落切片表

| 列 | 说明 |
|---|---|
| 原始文本 | 截断显示（超过 80 字截断），悬停展开全文 |
| 层级下拉 | H1 / H2 / H3 / H4 / H5 / Body，预填自动判定结果 |
| 置信度 | high（绿）/ medium（黄）/ low（红，高亮提示核验）|
| 命中规则 | 如"模块一·H2"、"模块四·拆分" |

- low 置信度行以红色背景高亮，目标核验工作量 ≤ 5%
- 下拉变更后实时触发 Tab3 对应段落更新；若变更段落属于上下文依赖的参考标题（如 `（一）` 类、模块二上层标题），则触发后续所有上下文相关段落的重新判定和级联更新，Tab1 层级下拉和 Tab3 预览同步刷新

### 4.2 Tab2 — 原文预览

使用 **mammoth** 库将原始 .docx 精确转为 HTML，保留原始排版（字体/字号/加粗/对齐/缩进等），只读展示，文档编码 UTF-8。
- .doc 文件先经 win32com 转为 .docx 后再由 mammoth 转 HTML
- mammoth 使用自定义样式映射，将 Word 段落格式属性映射为对应 CSS 内联样式

### 4.3 Tab3 — 格式化预览

后端按当前层级 + YAML 模板生成精确的带内联样式 HTML 片段，与 step4 输出的 .docx 排版效果保持一致（UTF-8 编码）：
- 字体、字号、加粗、对齐、首行缩进、行距均以 CSS 内联样式精确呈现
- HTML 渲染与 python-docx 写入属性严格对应，保证所见即所得
- Tab1 下拉变更后，发送增量更新请求，只刷新变更段落
- 用户点击"确认输出"→ 触发 step4

---

## 五、Step3：应用格式模板（内嵌于 step2）

- 从 `.yaml` 文件加载模板（路径可配置，内容不可修改）
- 模板统一存放于 `templates/` 目录（`report_cn.yaml` 移入该目录）；后端启动时扫描该目录下所有 `.yaml` 文件作为可用模板列表
- 前端页面顶部提供**模板选择下拉框**，列出所有可用模板；切换后 Tab3 实时用新模板重新渲染全部段落
- 当前默认模板：`templates/report_cn.yaml`（H1-H5 / Body，仿宋_GB2312，行距 1.5，页边距上下 2.54cm / 左右 3.17cm）
- Tab3 每次渲染即为 step3 的实时执行，HTML 渲染与 python-docx 写入属性严格对应（UTF-8 编码），用户确认时状态即为最终输出，无需重复计算

---

## 六、Step4：输出格式化文档

- `.docx` 输入 → 直接输出 `.docx`
- `.doc` 输入 → 用 `win32com.client` 调用本机 Word COM 转为 `.docx`，处理完再转回 `.doc`
- 表格/图片保留原始内容和格式，不做任何处理
- 输出文件名：`{原文件名}_formatted.{ext}`

---

## 七、技术栈

| 层 | 方案 |
|---|---|
| 后端 | Python + python-docx |
| Tab2 原文转 HTML | mammoth（自定义样式映射，精确保留原始排版）|
| doc 转换 | win32com.client（本机 Word COM，Windows 环境）|
| API | FastAPI |
| 前端 | 纯 HTML + JS |
| Tab3 实时预览 | 后端生成带内联样式的 HTML 片段返回前端 |
| 模板选择 | 前端顶部下拉框，后端扫描指定目录下所有 .yaml 文件 |

1. `report_cn.yaml` 路径可移动，内容不可修改
